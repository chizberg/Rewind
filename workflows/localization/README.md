# Localization workflows

Мульти-агентные workflow'ы для `Rewind/Localizable.xcstrings`. Заменяют скрипты
из `~/Documents/Rewind-localization`, которые дёргали `claude -p` по одной
строке через `subprocess`.

## Файлы

| Файл | Что это |
|---|---|
| `sync-translations.js` | Добить недостающие переводы + обновить stale/needs_review |
| `add-language.js` | Добавить новый язык целиком |
| `xcstrings.py` | Общий хелпер: inventory / apply / verify |

## Запуск

Workflow'ы запускаются через инструмент `Workflow` по пути к файлу:

```
Workflow({ scriptPath: "workflows/localization/sync-translations.js" })
Workflow({ scriptPath: "workflows/localization/add-language.js", args: "pl" })
```

Проще всего — попросить Claude Code словами: «прогони sync-translations» или
«добавь польский через add-language». Оркестрация многоагентная и платная, так
что запускается только по явной просьбе.

### Параметры (`args`)

Оба принимают объект; все поля необязательные:

```jsonc
{
  "file": "Rewind/Localizable.xcstrings",   // каталог
  "languages": ["de", "fr"],                // только sync: ограничить языки
  "language": "pt-BR",                      // только add-language (или args: "pt-BR")
  "chunkSize": 25,                          // только add-language
  "scratch": ".loc-workflow"                // временные файлы
}
```

## Как устроен `sync-translations`

```
Scan ──> Translate (Sonnet, 1 агент на язык)
              └──> Review (Opus + веб, 1 агент на язык)
                          └──[барьер]──> Apply ──> Report
```

- **Scan** — `xcstrings.py inventory` находит всё, что missing / stale /
  needs_review. Если пусто — workflow сразу выходит.
- **Translate** — Sonnet, по одному агенту на язык. Каждому агенту отдаётся
  *весь уже одобренный корпус этого языка* как эталон стиля.
- **Review** — Opus с `WebSearch`/`WebFetch`. Порог уверенности **0.8**: ниже —
  агент обязан идти в интернет (как Apple локализует этот же концепт, локализуется
  ли бренд в этой локали, грамматика), а не гадать. Он вправе переписать перевод.
- **Apply** — один агент, последовательно. Каталог — один файл, параллельные
  писатели его затрут.
- **Report** — markdown прямо в ответ, файлом не пишется.

`Translate` и `Review` идут пайплайном без барьера: язык уходит на ревью, как
только готов его драфт, не дожидаясь остальных.

## Как устроен `add-language`

```
Prepare ──> Research (Opus + веб: style guide + глоссарий)
              └──> Translate (Sonnet, по чанкам) ──> Review (Opus + веб)
                          └──[барьер]──> Harmonise ──> Apply ──> Report
```

Два отличия от `sync`:

- **Research** — отдельная фаза до перевода. Opus идёт в интернет и фиксирует
  конвенции локали: форма обращения (ты/вы), наклонение кнопок, капитализация,
  политика по брендам, типографика, глоссарий доменных терминов, грамматические
  ловушки. Результат — `style-guide.md`, обязательный для всех переводчиков.
  Без этой фазы каждый чанк изобретает свои конвенции.
- **Harmonise** — барьерная фаза после ревью. Чанки ревьюились изолированно,
  поэтому каждый локально корректен, а язык в целом может быть несогласован.
  Этот агент видит все чанки сразу и ловит дрейф терминологии, регистра,
  капитализации и разъехавшиеся парные строки (сортировки, типы карты, пары
  title/description в онбординге).

## Про «построчно»

Ты просил переводить построчно. Сделано **по языку** (в `sync`) и **чанками по
25** (в `add-language`), а не по одной строке на агента. Причина: 117 строк × 8
языков — это 936 агентов, что упирается в лимит в 1000 и стоит несоразмерно, а
главное — агент, видящий одну строку, не может быть согласован с остальными.
Половина ошибок в этом каталоге — не ошибки отдельной строки, а расхождения
между строками:

- `Date Ascending` → «Сначала старые» правильно только если знать, что это пункт
  пикера сортировки, а рядом есть `Date Descending`;
- `Google Street View` в одном месте «панорамы Google», в другом остаётся как
  есть — это решение уровня каталога, не строки;
- `Сравнить` (инфинитив) vs `Просматривайте` (императив) — конвенция регистра.

Батч по языку сохраняет «каждая строка переводится осознанно», но даёт агенту
контекст, без которого эти решения невозможны.

## Хелпер `xcstrings.py`

Работает и отдельно, без workflow'ов:

```bash
# что осталось перевести
python3 workflows/localization/xcstrings.py inventory Rewind/Localizable.xcstrings

# то же для одного языка + одобренный корпус как эталон стиля
python3 workflows/localization/xcstrings.py inventory Rewind/Localizable.xcstrings \
    --languages de --reference de

# применить патч (сначала всегда --dry-run)
python3 workflows/localization/xcstrings.py apply Rewind/Localizable.xcstrings \
    .loc-workflow/de.patch.json --dry-run

# проверить формат и покрытие
python3 workflows/localization/xcstrings.py verify Rewind/Localizable.xcstrings
```

Формат патча:

```json
{"language": "de", "updates": [{"key": "Paintings", "value": "Gemälde"}]}
```

`variationPath` (`"plural.one"`, `"device.iphone"`) — только для строк с
вариациями. `key` — всегда ключ каталога дословно: для символьных ключей вроде
`comparison_title` это сам ключ, а не английский текст.

### Что он гарантирует

- **Байт-в-байт формат Xcode.** Xcode пишет `.xcstrings` с разделителем `" : "`,
  отступом 2 и без финального перевода строки. Обычный `json.dump` переписывает
  весь файл; здесь `verify` проверяет, что сериализация совпадает с исходником
  ровно. Дифф содержит только реально изменившиеся строки.
- **Плейсхолдеры.** `apply` отклоняет перевод, у которого набор `%@` / `%lld` /
  `{name}` не совпадает с исходником.
- **Источник строки.** Для символьных ключей источник берётся из локализации
  `sourceLanguage`, а не из ключа.
- Пропускает строки с `shouldTranslate: false`.

## Отчёты

Отчёт **возвращается в чат, а не пишется файлом** — он одноразовый и не должен
попадать в коммит. Всегда содержит:

- что изменено — по **ключу и английскому тексту** одновременно;
- где ревьюер ходил в интернет, что нашёл, ссылки;
- **Open questions** — то, что не удалось решить, как чеклист. Вопросы уровня
  конвенций идут первыми: они влияют на все строки сразу.

## Проверка после прогона

```bash
python3 workflows/localization/xcstrings.py verify Rewind/Localizable.xcstrings
git diff --stat -- Rewind/Localizable.xcstrings
```

`roundTripsExactly: true` и дифф размером примерно в две строки на изменённый
перевод. Если переписан весь файл — что-то его переформатировало.

Черновики и патчи остаются в `.loc-workflow/` (в `.gitignore`) — по ним можно
разобрать, что решил переводчик и что переписал ревьюер.
